Lab Work

Laboratory Sessions

Progressive lab assignments building from basic UI to a fully functional Room-powered app.

Lab 1

User Interface Layout

Building the NextStop home screen using Jetpack Compose layouts

Reflection

In Lab 1, I learned how to use Column, Row, and Box composables to build a structured mobile interface inspired by the KITS app. It was my first time working with Jetpack Compose and I found the declarative approach very different from XML-based layouts. This lab gave me a strong foundation in understanding how Compose layouts work together to create a clean and responsive UI.

Lab 2

User Interaction & State

Adding TextField, Button, and mutableStateOf for dynamic input

Reflection

Lab 2 introduced me to state management in Jetpack Compose using mutableStateOf and remember. I added a Journey Search Bar where users can type a destination and tap a button to trigger a live response on screen. Understanding how remember keeps state alive across recompositions was the key learning moment in this lab.

Lab 3

Material Design & Animation

Applying Material Theme, Card composables, and expandable animations

Reflection

In Lab 3, I learned to apply a proper Material 3 theme by generating a colour palette using the Material Theme Builder at m3.material.io. I wrapped key sections in Card composables and added an expandable animation to service cards using AnimatedVisibility and animateContentSize. This lab taught me that professional theming is about using a system, not just picking colours manually.

Lab 4

Navigation & ViewModel

Multi-screen navigation with NavController and shared ViewModel state

Reflection

Lab 4 expanded NextStop into a multi-screen app using Jetpack Navigation Compose. I created a Routes object as a singleton to store all screen addresses and used NavController to navigate between six screens. The ViewModel allowed booking data to persist across screens using mutableStateOf and the copy() function on a data class.

Lab 5

Room Database

Implementing local data persistence with Room Entity, DAO, and Repository

Reflection

Lab 5 introduced Room Database to save booking history permanently on the device. I implemented the full architecture — Entity, DAO, Database singleton, Repository, and ViewModel — following clean separation of concerns. Using StateFlow and collectAsState() to observe the database and automatically update the UI was the most valuable concept I learned in this lab.

Projects

Mobile App Projects

Two iterations of NextStop — from a navigation-based booking app to a fully connected, hardware-aware transport platform.

Project 1

NextStop — Transport Booking App

Many Malaysians struggle to plan public transport because information is scattered across multiple platforms, causing people to choose private vehicles and increasing urban congestion. NextStop solves this by combining train, rail, and flight booking into one simple app supporting SDG 11.

Kotlin Jetpack Compose Navigation ViewModel Material 3 Room DB

Lessons Learned

Building Project 1 taught me how to structure a multi-screen app with clean navigation using NavHost and Routes. I learned that ViewModel is not just about storing data — it's about keeping data alive across screen transitions. The most important lesson was understanding how all layers (UI, ViewModel, Repository, Room) connect together in a proper Android architecture.

Project 2

NextStop — Fully Connected Transport Platform

Building on Project 1, NextStop needed cloud connectivity, live data, and hardware awareness to become a truly real-world transport app. Project 2 adds Firebase Firestore for community disruption reports, live weather via OpenWeatherMap API, and GPS-based nearby station detection.

Kotlin Jetpack Compose Room DB Firebase Firestore Retrofit GPS Sensor OpenWeatherMap

Lessons Learned

Project 2 was the most challenging and rewarding project of the semester. Integrating four technical pillars simultaneously taught me how modern Android apps are built in the real world — local persistence, cloud sync, REST API calls, and hardware sensors all working together. I learned that clean architecture is not optional when complexity grows; it is what makes the app maintainable and testable.
